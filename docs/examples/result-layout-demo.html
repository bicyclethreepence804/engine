Open Core engine - output map and developer guide

Deterministic JSON from @kiploks/engine-core and how it relates to product-style blocks. Use the tabs below for setup instructions, each public entrypoint, and contract reference. This file is static documentation, not the production UI.

1. Work inside the engine repository (contributors)

From the engine folder (Open Core git root):

cd engine
npm install
npm run build
npm run engine:examples:generate-samples

After npm run build, run npm run engine:examples:generate-samples to refresh docs/examples/sample-output/*.json so they match metadata.engineVersion from the built packages. If the static demo must match those files byte-for-byte, update the embedded SAMPLE_MINIMAL and SAMPLE_WFA objects in this file. The Monte Carlo tab uses frozen SAMPLE_PATH_MC and SAMPLE_PRO_MONTE; update those when path MC or professional Monte outputs change (see monte-carlo-example.md and WFA docs).

2. Consume published packages (integrators)

npm install @kiploks/engine-core @kiploks/engine-contracts
  • Types: AnalyzeInput, AnalyzeOutput, TradeBasedWFAInput, PrecomputedWFAInput, WFAAnalysisOutput from @kiploks/engine-contracts.
  • API: analyze(), analyzeFromTrades(), analyzeFromWindows() from @kiploks/engine-core.

Minimal TypeScript

import { analyze } from "@kiploks/engine-core";

const result = analyze(
  { trades: [{ profit: 0.05 }, { profit: -0.02 }] },
  { seed: 42, decimals: 8 },
);
console.log(result.summary, result.metadata);

4. CLI (@kiploks/engine-cli)

npx kiploks --help
npx kiploks validate <file.json> --schema trade-based-wfa --explain
npx kiploks analyze <input.json> --json --seed 42 --decimals 8
npx kiploks analyze-trades <trades.json> --format raw --json \
  --in-sample-months 2 --out-of-sample-months 1 --step rolling

Install CLI globally or as a dev dependency if you prefer a local kiploks binary.

Expected in this sample: verdict ROBUST and score 0 (not a bug)

              

                At a glance this pair can look wrong. It is intentional: two independent pipelines fill different
                fields.
              

              
    
                
  1. 
                  WFE verdict (wfe.verdict, e.g. ROBUST) comes only from
                  rank-WFE and permutation p-value heuristics. It does not set the numeric robustness score.
                
    2. 
                
  2. 
                  Robustness score (robustnessScore) is computed by
                  calculateRobustnessScoreFromWfa (four modules: validation, risk, stability,
                  execution). Public WFA passes an imputed input. With no parameter-sensitivity data,
                  the stability module is 0; blocked modules force the final score to
                  0 (see core/src/robustnessScoreFromWfa.ts).
                
    3. 
                
  3. 
                  A full product TestResultData build can supply real module inputs and then
                  the score can be non-zero while WFE verdict stays as computed.

        The methodology page in the product (e.g. /methodology) documents
        Walk-Forward Efficiency bands on the median OOS/IS ratio (PASS / ACCEPTABLE /
        WARN / FAIL) and a separate Final Verdict (ROBUST / CAUTION / DO NOT DEPLOY). The Open Core
        JSON field wfe.verdict is a different label set: rank-WFE thresholds
        from verdictFromRankWfe in the engine. Do not mix them up when reading this demo.
      
    
      
    
        
    1. Open Core JSON: wfe.verdict (rank-WFE heuristic)
    
        
    
          Source: core/src/wfa/wfeCalculator.ts (verdictFromRankWfe).
          Based on rankWfe (mean of OOS rank / IS rank per window), not on the median OOS/IS ratio table
          below.
        
    
        
    
          
            
    
              Condition (rankWfe)
              wfe.verdict
              Note
            
    
              
          
            
    
              rankWfe ≥ 1.15
              ROBUST
              Strong rank transfer (heuristic).
            
    
            
    
              1.06 ≤ rankWfe < 1.15
              ACCEPTABLE
              Middle band on rank scale.
            
    
            
    
              1.0 ≤ rankWfe < 1.06
              WEAK
              Low rank transfer.
            
    
            
    
              rankWfe < 1.0
              FAIL
              
                Also forced FAIL in buildWfeResult when mean IS > 0 and mean OOS < 0.
              
            
    
              
        
    
        
    
          Badge colors here mirror the product palette for teaching only. Labels
          ROBUST / ACCEPTABLE / WEAK in this column are not the methodology PASS/WARN
          bands and not the Final Verdict row.
        
    
      
    
      
    
        
    2. Product methodology: Walk-Forward Efficiency (median OOS/IS)
    
        
    
          Same threshold copy as the Kiploks methodology page Walk-Forward section (reference UI).
          This describes the efficiency ratio, not wfe.verdict in Open Core
          JSON.
        
    
        
    
          
            
    
              Range (median OOS/IS)
              Label
              Meaning
            
    
              
          
            
    
              ≥ 0.7
              PASS
              Robust carry-over; the strategy "behaves" as expected.
            
    
            
    
              0.5 – 0.7
              ACCEPTABLE
              Adequate stability; monitor for alpha decay.
            
    
            
    
              0.2 – 0.5
              WARN
              Fragile; significant performance drop Out-of-Sample.
            
    
            
    
              < 0.2
              FAIL

              —

    3. Product methodology: Final Verdict (report summary)
    
        
    
          End-of-report verdict from multiple gates (robustness score, DQG, WFE, risk, costs). Not stored as
          wfe.verdict in the minimal Open Core contract.
        
    
        
    
          
            
    
              Verdict
              Meaning
            
    
              
          
            
    
              ROBUST
              Strategy passes validation, risk, and cost checks. Review the full report and consider deploying.
            
    
            
    
              CAUTION
              Some gates failed or are borderline. Fix the flagged issues before deploying.
            
    
            
    
              DO NOT DEPLOY
              
                Critical failures detected (e.g. data quality, WFE, or risk). Address all summary items before any
                deployment.
